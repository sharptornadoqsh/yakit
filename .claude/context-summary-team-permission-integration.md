# 项目上下文摘要（团队权限集成）

生成时间：2026-07-23 13:50:00 +08:00

## 一、相似实现分析

- `online/online/server/api/collaboration_admin.go`
  - `/v2/me` 返回当前用户及按团队划分的 `memberships`。
  - 每项成员身份包含团队、角色、权限代码和可访问项目。
  - 团队列表仅返回团队基础字段，不能直接用于当前用户权限判定。
- `yakit-website-master/yakit-website-master/src/admin/permission/permissionCore.ts`
  - 以团队编号为键建立权限索引。
  - 直接使用服务端点分隔权限代码，并将 `administrator` 角色映射为通配权限。
- `yakit-website-master/yakit-website-master/src/admin/permission/permission.tsx`
  - 页面初始化时调用当前用户接口，再将成员身份转换为当前团队权限。
  - 权限接口失败时不授予额外权限。
- `yakit/app/renderer/src/main/src/pages/teamCollaboration/TeamCollaborationPage.tsx`
  - 当前实现只调用团队列表，却从团队对象读取角色和权限。
  - 当前写权限代码使用冒号分隔格式，与服务端 `project.manage`、`test_data.write`、`test_result.write` 不一致。
  - 单个 `canWrite` 同时控制项目管理、归档发布、测试数据和测试结果，未反映服务端的细粒度权限。
  - 团队成员的用户和角色来自嵌套 `user`、`roles` 字段，项目成员仅携带用户编号，现有扁平字段读取无法显示真实成员信息。

## 二、项目约定

- React 函数组件使用大驼峰命名，函数与状态使用小驼峰命名。
- TypeScript 采用两个空格、单引号、无分号和尾随逗号。
- 网络请求统一复用 `teamCollaboration.ts` 中的服务函数。
- 接口响应通过现有 `getList` 与 `getValue` 兼容响应包装和字段差异。

## 三、可复用组件

- `getMe`：读取当前用户的团队成员身份、角色和权限。
- `listTeams`：读取当前用户可见团队。
- `getList`、`getValue`、`getId`：规范化接口响应和实体字段。
- `TeamCollaborationPage.test.tsx`：页面权限和数据操作的现有组件测试。

## 四、测试策略

- 使用现有 Vitest 与 Testing Library。
- 团队列表模拟为仅含基础字段，确保测试能够复现真实服务端响应。
- `/v2/me` 模拟只读成员、具备多项写权限的成员、管理员角色和单项写权限成员。
- 验证项目管理、归档发布、测试数据和测试结果按钮分别受对应权限约束。
- 团队成员与项目成员使用服务端真实嵌套响应，验证用户名和角色名称。
- 只执行相关单元测试、格式检查与差异检查，不执行 Yakit 编译或构建。

## 五、依赖和集成点

- 外部依赖没有变化。
- 权限来源为在线服务 `/v2/me`。
- 团队与成员身份通过团队编号关联。
- 服务端权限代码为点分隔格式。
- 归档发布同时需要项目管理与测试数据写入权限。

## 六、技术选择

- 保留团队列表作为团队展示来源，以当前用户成员身份作为权限来源。
- 在页面数据加载阶段合并团队和成员身份，不修改服务端团队列表协议。
- 按操作类型分别计算权限，保持界面状态与服务端鉴权一致。
- 管理员角色使用现有管理网站的通配权限规则。

## 七、风险

- 当前用户接口失败时不能可靠确定写权限，因此页面应显示请求错误且不授予写权限。
- 自定义角色可能只拥有部分写权限，必须避免把单项权限扩大为全部写权限。
- 团队编号可能采用数字或字符串表示，关联时统一转为字符串。
