# 外部账号服务集成说明

## 一、范围与边界

本阶段复用仓库外已部署的账号服务，客户端负责服务地址配置、账号登录、当前用户状态、用户管理、角色与权限管理及会话失效反馈。仓库内没有新增账号服务、模拟数据或替代 API。

阶段五基于单仓源码得出的“服务端实现未在本仓库内”结论仍属当时证据；本阶段通过已启动服务与客户端开发态验证，证明该前端已有可用的外部服务契约。该结论不覆盖团队工作区、共享项目、服务端安全总览或多客户端协作。

## 二、地址与 HTTP 契约

- 用户保存服务根地址，客户端统一生成以 `/api/` 结尾的请求基址。
- 根地址无论是否自带 `/api`，请求基址都只保留一次 `/api/`，并移除查询参数与片段。
- 登录使用 `POST /api/urm/login`，请求体字段为 `user_name` 与 `pwd`。
- 用户使用 `/api/urm`、`/api/urm/edit`、`/api/urm/reset/pwd`。
- 角色使用 `/api/roles` 与 `/api/roles/detail`。
- 标准响应结构为 `{code, message, data}`。主侧 HTTP 层读取该结构后，把 `data` 交给页面调用方。
- 登录成功后，服务返回的令牌原值写入 `Authorization` 请求头，不添加 `Bearer` 前缀。
- 服务地址发生变化时，主侧清空用户、角色、令牌及登录标志，并通知渲染端退出原会话。

关键实现位于 `app/main/httpServer.js`、`app/main/state.js`、`app/main/handlers/userInfo.js` 与 `app/renderer/src/main/src/components/ConfigPrivateDomain/ConfigPrivateDomain.tsx`。

## 三、登录与会话

登录表单先用客户端编解码能力生成密码的 MD5 值，再提交账号名与摘要。登录成功后，`companySignIn` 保存用户标识、角色、显示名和令牌，渲染端状态随之更新。

本阶段修正了两项会话状态问题：标准服务响应不再被多包一层 `data`；服务地址变化或本地退出时会清空完整用户状态。远端退出请求失败时，本地退出仍会完成。HTTP 状态或服务业务码为 `401` 时，既有请求层会触发本地退出提示和登录态清理。

配置中若保存密码，当前实现写入可逆的 Base64 编码。它避免配置文件出现原始明文，但不构成密码学保护。服务端密码存储方式没有材料，本阶段不作结论。

## 四、用户管理

“用户管理”页面复用现有账号与组织页面，具备搜索、组织筛选、创建、编辑、删除、批量删除与密码重置能力。页面只允许已登录管理员访问；未登录或非管理员访问时显示统一无权限状态。

开发态验证读取到一条服务账号记录及其组织与角色信息，证明列表请求、组织展示和角色展示可用。为避免改变外部服务数据，本阶段没有触发创建、编辑、删除或密码重置。

## 五、角色与权限

“角色与权限”页面复用现有角色列表、详情、创建、编辑和删除能力。页面只允许已登录管理员访问。内置角色不会纳入批量删除集合，提交前仍会再次剔除内置角色。

权限列改为读取服务返回的 `checkPlugin` 字段，不再依据角色名称推断。角色详情中的插件选择值统一为插件标识；保存时分别生成插件类型与插件标识，避免对象被转换为无效字符串。

开发态验证读取到三类内置角色。本次服务数据未给这些角色授予插件操作权限，因此权限列显示 `-`。当前只有管理员账号可用，多账号权限差异尚未动态验证。

## 七、验证证据

开发态主窗口使用用户已启动的服务完成以下验证：

- 服务登录成功，主侧状态显示已登录、平台为企业服务、角色为管理员，用户标识与令牌均存在。
- 用户管理页面显示组织结构、账号列表、角色、创建时间与管理操作。
- 角色与权限页面显示三条内置角色记录、权限列与创建时间。
- 管理员菜单显示“用户管理”和“角色与权限”，用户信息区域显示管理员角色。
- 页面未见致命脚本异常或横向溢出。

截图保存在本机临时目录，未纳入仓库。动态验证没有记录真实密码、完整令牌或服务内账号详情。

## 八、验证限制

- 没有可用的非管理员账号，因此无权限页面与多角色权限差异只有源码证据。
- 没有主动制造离线、错误密码或令牌过期，以免干扰用户当前会话；相关分支只有源码证据。
- 没有改变服务地址来验证旧会话清理；该项由主侧状态代码支撑。
- 没有改变外部服务中的用户或角色数据。
- 没有启动全量测试、全量类型检查、全量规则检查、构建、打包或跨平台验证。
